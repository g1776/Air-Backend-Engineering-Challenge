During this 4 hour process, I took an interative approach to designing this system. I approached the problem in this order:

1. Solve the batching issue. I knew this was the core of the problem and wanted to get it out of the way first. Splitting the event ingestion and batch resolution up into two steps was key to keeping the write path simple and low-cost.
2. Add the deadline watch functionality. This was a natural extension of the batching system, and I was able to reuse much of the same infrastructure for managing state and scheduling notifications. The main change was extending the data model to support watches and adding logic for creating, canceling, and firing them based on time.
3. I then revisited the grouping logic for batch generation and realized that the batch key logic was too rigid and simplistic for the requirements. I updated the batch key to be more flexible and support multiple dimensions of grouping, which allows for more powerful and customizable notification rules.
4. With core functionality in place, I added in notification preferences. I understood from the beginning that this would be an administrative task - stopping processes in their tracks - so I waited until the main flow was solid before adding this. This involved adding a new data model for user preferences, updating the batch finalization logic to check preferences before generating notifications, and ensuring that the API could handle preference updates in real time.
5. Liking the state of the system, I thought about scaling and extensibility. What if we support new concepts beyond Assets and Boards? What if we want to add new channels or notification types? I made sure the design was modular and decoupled enough to support future growth without major redesign.
